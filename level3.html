Level 3: AI in Your Terminal

At Level 3, the AI runs in your terminal. You describe a goal; it reads your files, writes code, runs scripts, and iterates on failures until the task is done. You review the result. What makes this more capable than an editor assistant is not what the agent can see; it is that it can execute. In the editor, you run the tests and bring errors back to the AI. Here, the agent runs that loop itself.

Most terminal AI tools require a paid subscription. Free tiers typically provide access to web-based chat only — not the terminal agent. Check the pricing for whichever tool you use before getting started.

A terminal agent is the right tool when the task requires execution: running tests, iterating across files, working through a multi-step process. For inline suggestions while you’re actively writing code, the IDE is still the natural fit. The two complement each other; most people use both.

The new development cycle

Step In the editor In the terminal
Organize Describe the goal in chat; AI proposes structure Describe the goal; AI reads your existing files and proposes or creates structure
Write AI suggests edits inline; you accept or reject AI writes to files directly; you review what changed
Test You run tests and bring failures back to AI AI runs tests, reads the output, and iterates on failures itself
Share AI writes a commit message from your diff; you commit AI can stage, write the message, and commit; you review before or after
Deploy Not covered Scripted pipelines can be handed to the agent; it runs them and reports the result

The interaction loop

Every terminal AI agent works the same way: you describe a goal, the agent reads your files and environment, acts, reads the output, and reports back. You review and continue or redirect.

Understanding this loop changes how you write prompts. You are not giving instructions to someone who will ask for clarification — you are describing a goal for an agent that acts immediately. Keep goals narrow enough to review comfortably.

Your project

Context engineering is the new prompt engineering.

The agent only has access to what you point it at. Your folder structure is not just organization — it defines what the agent can see, and specific files carry your standing instructions between sessions. A well-structured project makes the agent faster and more reliable from the first prompt.

my-project/
├── context.md         ← standing instructions, read every session
├── progress.md        ← handoff notes between sessions
├── docs/
│   └── plans/         ← implementation plans
└── src/

The context file

Every session starts cold. The agent has no memory of previous sessions — no knowledge of your project structure, conventions, or the decisions you made last week. A context file, a markdown file in your project root, is how you fix that. Write your standing instructions there once; the agent reads it at the start of every session. Keep it factual: project structure, conventions, constraints that apply to every task. Specifics for the current task go in the prompt.

The folders you list and the directories you grant the agent access to form the same boundary. Start narrow: read and write within the project directory, run test and build commands you already use, no broader filesystem access. An agent with write access to your filesystem and permission to run arbitrary commands can make changes you didn’t intend. Review what it’s done at each step, especially early on.

The context window

The context window is the total amount of text — instructions, files, code, and conversation history — that the model can hold at once. As a session runs, everything accumulates there: your instructions, the files the agent read, the code it wrote, the test output, your feedback. This window has a fixed size, passed back and forth with every message. Once it fills, the agent starts dropping the oldest entries to make room.

In practice, a session that starts well can become unreliable as context fills. The agent forgets constraints you set earlier, duplicates code it already wrote, or loses track of decisions. This is not a defect in any specific tool — it is a fundamental property of how these models work.

Most tools offer a compaction command to summarize the current context and continue without restarting. This sounds useful but usually isn’t. Compaction compresses the window, but degraded reasoning from before the compaction still affects the session. Starting fresh with a written handoff produces better results.

Friends don’t let friends compact.

Between sessions

End sessions before context degrades. Most tools display context usage somewhere, and if yours doesn’t, it is worth making it visible. When context approaches 50%, end the session:

Write a summary of what we've done and what's left to progress.md

The next session reads that file and continues cleanly. Treat sessions as bounded units of work — end when the immediate task is done, not when the window is full. Avoid open-ended prompts like “clean up the whole codebase.” Break large goals into specific, bounded tasks.

Planning

Ad-hoc prompts produce worse results than structured ones. A structured workflow replaces guessing with agreement: describe the goal, clarify requirements, produce a plan in docs/plans/, then execute against it. Each phase is short. The execution is faster and produces fewer unintended changes.

Scope explicitly when executing. Vague file access reliably produces unintended changes — name the files and constraints before the agent starts writing.

MCP

Model Context Protocol is an open standard for connecting AI agents to external tools. Instead of pasting documentation into a prompt or building a one-off integration script, you configure an MCP server once and the agent can call it in any session.

An MCP server is a small program the agent invokes natively, indistinguishable from a built-in capability once configured. Common uses: pulling current library documentation, querying a database, running a custom analysis, accessing an API. Configure it once; it is available in every session.

Try it out with Claude Code

The concepts above apply to any terminal AI tool. Claude Code is one of them. If you want to try it, this section covers an efficient setup: the tool itself plus seven integrations that implement what we’ve discussed. A project sandbox and quick reference table follow below.

Install Claude Code

Navigate to your project and type claude to start a session. Many developers keep their IDE open to review files while running Claude Code in a terminal alongside it. Claude makes the changes; the IDE shows you what changed.

Sandbox

Create a dedicated folder anywhere on your computer and drop in this structure. It gives you a clean place to experiment without touching any existing project — open it in your terminal and you are ready to go.

my-project/

├── docs/
│   └── plans/                 ← implementation plans go here

01 — Statusline

Make context usage visible: the most critical metric in a session

The statusline adds a persistent bar at the bottom of your terminal showing context window percentage, session cost, model name, elapsed time, and git status. Without it, you notice context degradation when Claude starts making mistakes. With it, you can end sessions before quality drops.

Find install instructions in the Claude Code documentation or by asking Claude: How do I install the statusline?

Windows note: The setup wizard sometimes fails on Windows. If the guided setup does not complete, install manually: npx cc-statusline@latest in your terminal.

02 — Superpowers

Structured sub-agent workflows: the highest-impact single install

Superpowers is a Claude Code plugin that implements the planning and sub-agent workflow described above. Instead of one session accumulating context until it degrades, your main Claude instance dispatches focused sub-agents — each with its own fresh context window. The orchestrator receives summaries, not full workloads.

Superpowers enforces the brainstorm → plan → execute workflow: clarify requirements before writing anything, produce a spec, execute against it.

Install via the Claude Code plugin menu — type /plugin inside a session and find Superpowers in the list.

If you install one thing from this page, install Superpowers.

03 — Sequential Thinking

Structured reasoning before complex tasks

Sequential Thinking is an MCP server that adds explicit chain-of-thought reasoning to Claude Code. Before acting on complex tasks, Claude works through a visible step-by-step reasoning process — understand the problem, identify constraints, consider approaches, then implement.

Ask Claude to install it: Please install the sequential thinking MCP server

04 — Context7

Current library documentation: eliminates hallucinated APIs

Claude Code’s training data has a cutoff date. When you ask it to use a library, it works from training data that may be months out of date — confidently using deprecated methods or renamed APIs. Context7 pulls current, version-specific documentation from live repositories and injects it into Claude’s context at prompt time.

Install via the Claude Code plugin menu — type /plugin inside a session and find Context7 in the list.

05 — Warp

A better terminal for running Claude Code

Warp is a modern terminal — a replacement for Windows Terminal, Mac Terminal, or the VS Code integrated terminal. You run Claude Code inside Warp.

Split panes let you run Claude on one side and browse your project files on the other. When Claude writes a plan to a file, you can read it while Claude is still working. Each command and its output is a discrete block; no scrolling through walls of output to find what Claude ran.

Download from Warp — available for Mac, Linux, and Windows. Free to use.

06 — Happy Engineering

Full terminal access from your phone

Happy Engineering runs a live terminal session on your computer that you control from your phone’s browser. Setup takes about a minute — no accounts, no configuration, simpler than any remote-access alternative. When you connect from your phone, you are controlling the actual terminal on your machine — every integration you have configured is available. Tasks you start on your phone continue running on your computer.

Install from Happy Engineering. Free and open source.

07 — Skills

Reusable workflows for tasks you do repeatedly

Skills are markdown files in .claude/skills/ that Claude Code reads when you invoke them. Instead of re-explaining a workflow every session, you encode it once and call it by name.

.claude/
  skills/
    add-module.md    ← invoked with /add-module
    run-analysis.md  ← invoked with /run-analysis

Describe the repetitive task in detail and ask Claude to build the skill for you. After a few iterations, it becomes a reliable automation available in every session.
