*Dedicated coding agent — AI in your terminal with full environment access*

At Level 3, the AI moves out of the editor and into your terminal. It can read files, write code, run scripts, install packages, and iterate on its own until a task is complete. Seven integrations make it significantly more capable.

> **Paid subscription required.** Claude Code requires an Anthropic account with a paid plan ($20/month Pro). It is not available on the free tier.

> **Claude Code is the example on this page.** Other terminal-level tools exist — Aider, Cursor agent mode, Gemini CLI, and others follow the same principles. The integrations below are Claude Code-specific, but the workflow concepts apply across tools.

## Before and after {#before-after}

| | Level 2 | Level 3+ |
|---|---|---|
| **Where AI lives** | Inside your editor | In your terminal |
| **What AI can see** | Open files, project folder | Everything on your machine |
| **What AI can do** | Propose changes, generate code | Run code, install packages, read output, iterate |
| **Your role** | Review diffs, accept or reject | Describe the goal, come back to results |
| **The constraint** | AI suggests but cannot execute | No constraint — AI acts |

The five-step workflow is still there. What changes is who does the manual steps. At Level 2, you run tests and copy the output back into chat. At Level 3, the agent runs the tests itself and iterates on the failure without you in the loop.

## Getting started {#getting-started}

Install Claude Code and find platform-specific setup instructions at the official terminal guide:

**[code.claude.com/docs/en/terminal-guide#windows](https://code.claude.com/docs/en/terminal-guide#windows)**

Once installed, navigate to your project and type `claude` to start a session. You talk to it in plain English. It reads your files, writes code, runs scripts, sees the output, and reports back.

**Example session:**

*You type:*

Read my project and summarize what it does and how it’s structured.


*Claude does:* Reads every file in the directory and gives a plain-English summary of the project's structure, purpose, and main components.

*You type:*

Write a function in analysis.py that groups sales by region and returns a sorted DataFrame. Then write a test for it and run it.


*Claude does:* Writes the function, writes the test, runs it, reads the output, fixes any failure, and reports the result.

> **Claude Code alongside your IDE.** Many developers keep VS Code open to read and review files while running Claude Code in a terminal alongside it. Claude makes the changes; the IDE shows you what changed. They complement each other.

## The core problem: context {#context}

Before installing any integrations, understand the most important concept for working effectively with Claude Code: the context window.

When a session starts, Claude knows nothing about your project. As you give commands, all of that information accumulates in a context window — a fixed-size memory passed back and forth with every message. Once it's full, Claude starts forgetting earlier entries to make room for new ones.

In practice, a session can start sharp and become unreliable as context fills — forgetting constraints you set earlier, writing duplicate code, losing track of decisions already made. This is a fundamental property of how these models work.

> **The rule: don't let context go past ~50%**
>
> The statusline (integration 01 below) shows the exact percentage. When it approaches 50%, end the session and start fresh. Before ending:
>
> ```
> Write a summary of what we've done and what's left to progress.md
> ```
>
> The next session starts by reading that file.

## 01 — Statusline {#statusline}

*Make context usage visible — the most critical metric in a session*

The statusline adds a persistent bar at the bottom of your terminal showing context window percentage, session cost, model name, elapsed time, and git status. Without it, you only notice context degradation when Claude starts making mistakes. With it, you can end sessions before quality drops.

Find install instructions in the Claude Code documentation or by asking Claude: `How do I install the statusline?`

> **Windows note:** The setup wizard sometimes fails on Windows. If the guided setup doesn't complete, install manually: `npx cc-statusline@latest` in your terminal.

> **On /compact: don't rely on it.** Claude Code has a `/compact` command that summarizes the conversation and starts fresh. It's tempting but usually counterproductive — some degraded context remains after compaction. A clean session start with a written summary produces better results.

## 02 — Superpowers {#superpowers}

*Structured sub-agent workflows — the highest-impact single install*

Superpowers is a Claude Code plugin that enables sub-agent workflows: instead of one session accumulating context until it degrades, your main Claude instance acts as an orchestrator and dispatches specialized sub-agents to do focused work — each in its own fresh context window. Code writer, code reviewer, test writer — each starts fresh. The orchestrator receives summaries, not full workloads.

Superpowers also enforces a planning workflow: brainstorm first, write a spec, implement from the spec. This alone produces significantly better output quality than ad-hoc requests.

Install via the Claude Code plugin menu — type `/plugin` inside a session and find Superpowers in the list.

> If you install one thing from this page, install Superpowers. The shift from ad-hoc requests to plan-driven development with sub-agents is the biggest quality jump available in Claude Code.

---

## 03 — Sequential Thinking {#sequential-thinking}

*Deeper reasoning for complex problems*

Sequential Thinking is an MCP server that adds structured chain-of-thought reasoning to Claude Code. Before acting on complex tasks, Claude works through a visible step-by-step reasoning process — understand the problem, identify constraints, consider approaches, then implement. You can catch wrong reasoning before it produces wrong code.

**What MCP means:** Model Context Protocol is an open standard that lets Claude Code connect to external tools as built-in capabilities. You configure an MCP server once; it's available in every session.

Ask Claude to install it for you: `Please install the sequential thinking MCP server`

---

## 04 — Context7 {#context7}

*Up-to-date library documentation — eliminates hallucinated APIs*

Claude Code's training data has a cutoff date. When you ask it to use a library, it works from training data that may be months out of date — confidently using deprecated methods or renamed APIs. Context7 fixes this. It's a Claude plugin that pulls current, version-specific documentation from live repositories and injects it into Claude's context when you ask about a library.

Use it by adding `use context7` to any prompt involving a library:

Build a FastAPI endpoint that validates JSON input. use context7


Install via the Claude Code plugin menu — type `/plugin` inside a session and find Context7 in the list.

## 05 — Warp {#warp}

*A better terminal for running Claude Code*

Warp is a modern terminal application — a replacement for Windows Terminal, Mac Terminal, or the terminal built into VS Code. You run Claude Code *inside* Warp.

The key features for Claude Code use: split panes let you run Claude on one side and browse your project files on the other. When Claude writes a plan to a file, you can read it while Claude is still working. Each command and its output is a discrete block — no scrolling through walls of output to find what Claude ran.

Download from [warp.dev](https://warp.dev) — available for Mac, Linux, and Windows. Free tier covers standard use.

---

## 06 — Happy Engineering {#mobile}

*Full terminal access from your phone*

Happy Engineering runs a live terminal session on your computer that you control from your phone's browser. When you connect from your phone, you are controlling the actual terminal on your machine — Superpowers, Context7, everything you've configured is available. Tasks you start on your phone continue running on your computer.

Install from [happy.engineering](https://happy.engineering). Free and open source.

---

## 07 — Skills {#skills}

*Custom reusable workflows for tasks you do repeatedly*

Skills are markdown files in `.claude/skills/` that Claude Code reads when you invoke them. Instead of re-explaining a process every session, you encode it once and call it with `/skill-name`.

.claude/ skills/ add-module.md ← invoked with /add-module run-analysis.md ← invoked with /run-analysis


Build skills using Superpowers — describe the repetitive task in detail and ask it to create the skill for you. After a few feedback iterations, the skill becomes a reliable automation available in every session.

## Prompting and workflow {#prompting}

Level 3 requires different habits than Level 2. At Level 2, a vague prompt produces a suggestion you can reject. At Level 3, a vague prompt can produce file changes across your project before you realize the direction is wrong.

**Scope every request explicitly.**

❌ “Clean up my codebase” ✅ “Refactor the error handling in src/pipeline.py only. Don’t change function signatures. Don’t touch any other files.”


**CLAUDE.md — persistent session context.**

`CLAUDE.md` is a markdown file in your project root that Claude Code reads at the start of every session. Write your standing instructions here once — project structure, conventions, what to avoid. Every session starts with that context already loaded.

```markdown
# Project context

Python data pipeline. Use pandas for all data work.
Every new function needs a docstring and a test in tests/.
Never modify files in data/raw/.

Skills — explicit invocation.

Skills do not activate automatically unless you configure them to in CLAUDE.md. Invoke them by name (/skill-name). To have Claude use a skill automatically on a category of task, add the instruction to CLAUDE.md:

# In CLAUDE.md
Before implementing any new feature, use the brainstorming skill.

To exclude a skill from a specific task: “Don’t use the debugging skill for this — just make the fix.”

Keep sessions short.

End sessions when the task is complete, not when context is full. A clean session start with a written summary always produces better results than continuing a degraded session.
