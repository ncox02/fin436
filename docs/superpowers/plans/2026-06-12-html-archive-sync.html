    <blockquote style="border-left:3px solid var(--accent2);padding-left:1rem;font-style:italic;margin:0 0 1.5rem;"><strong>Context engineering is the new prompt engineering.</strong></blockquote>
    <p>The agent only has access to what you point it at. Your folder structure is not just organization &mdash; it defines what the agent can see, and specific files carry your standing instructions between sessions. A well-structured project makes the agent faster and more reliable from the first prompt.</p>

    <div class="code-block">my-project/

    <h3 class="sub-section-title">The context file</h3>
    <p>Every session starts cold. The agent has no memory of previous sessions &mdash; no knowledge of your project structure, conventions, or the decisions you made last week. A context file, a markdown file in your project root, is how you fix that. Write your standing instructions there once; the agent reads it at the start of every session. Keep it factual: project structure, conventions, constraints that apply to every task. Specifics for the current task go in the prompt.</p>
    <p>The folders you list and the directories you grant the agent access to form the same boundary. Start narrow: read and write within the project directory, run test and build commands you already use, no broader filesystem access. An agent with write access to your filesystem and permission to run arbitrary commands can make changes you didn&rsquo;t intend. Review what it&rsquo;s done at each step, especially early on.</p>

    <h3 class="sub-section-title">The context window</h3>
    <p>The context window is the total amount of text &mdash; instructions, files, code, and conversation history &mdash; that the model can hold at once. As a session runs, everything accumulates there: your instructions, the files the agent read, the code it wrote, the test output, your feedback. This window has a fixed size, passed back and forth with every message. Once it fills, the agent starts dropping the oldest entries to make room.</p>
    <p>In practice, a session that starts well can become unreliable as context fills. The agent forgets constraints you set earlier, duplicates code it already wrote, or loses track of decisions. This is not a defect in any specific tool &mdash; it is a fundamental property of how these models work.</p>
    <p>Most tools offer a compaction command to summarize the current context and continue without restarting. This sounds useful but usually isn&rsquo;t. Compaction compresses the window, but degraded reasoning from before the compaction still affects the session. Starting fresh with a written handoff produces better results.</p>
    <div class="callout">
      <div class="callout-icon"></div>
      <div>
        <strong>Friends don&rsquo;t let friends compact.</strong>
      </div>
    </div>

    <h3 class="sub-section-title">Between sessions</h3>
    <p>End sessions before context degrades. Most tools display context usage somewhere, and if yours doesn&rsquo;t, it is worth making it visible. When context approaches 50%, end the session:</p>
    <div class="code-block">Write a summary of what we've done and what's left to progress.md</div>
    <p>The next session reads that file and continues cleanly. Treat sessions as bounded units of work &mdash; end when the immediate task is done, not when the window is full. Avoid open-ended prompts like &ldquo;clean up the whole codebase.&rdquo; Break large goals into specific, bounded tasks.</p>

    <h3 class="sub-section-title">Planning</h3>
    <p>Ad-hoc prompts produce worse results than structured ones. A structured workflow replaces guessing with agreement: describe the goal, clarify requirements, produce a plan in <code>docs/plans/</code>, then execute against it. Each phase is short. The execution is faster and produces fewer unintended changes.</p>
    <p>Scope explicitly when executing. Vague file access reliably produces unintended changes &mdash; name the files and constraints before the agent starts writing.</p>
