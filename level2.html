AI in Your Editor

Try It Prompting Editor & Context callout? Level 3
The shift

This is not a faster version of copy-paste

The old way — browser chat

The copy-paste loop

  • Switch tabs between editor and browser dozens of times per session
  • Manually copy code into chat, losing the surrounding context
  • Paste output back into the editor, often needing manual cleanup
  • Copy terminal errors out, paste them back into the chat
  • Every new chat starts from scratch — no memory of your project
  • Suggestions frequently conflict with code the AI hasn't seen
Level 2 — in-editor AI

AI where your code lives

  • Chat panel lives inside your IDE — no window switching
  • AI reads your actual open file before you ask anything
  • Changes proposed inline, directly in the file you're editing
  • Side-by-side diff shown before anything is touched — you decide
  • Select any block and ask for targeted edits in place
  • Suggestions grounded in your real code, not a description of it
Try it

10 Simple Tasks with AI

Each exercise is self-contained. Do them in any order, skip what doesn't interest you, come back to others later. Exercises 4 through 8 share the same code block if you want to see several capabilities working on the same function — but none of them depend on completing the others first.

1
Organize
Propose a project structure
See AI reason about structure before a single file exists

Open the chat panel in your editor with no files open. Send the prompt below. Read what comes back — does the structure make sense for the project? Push back in the chat if not. When you're satisfied, ask it to list the terminal commands to create the folders and files.

I'm building a Python project that reads a folder of CSV files, cleans the data, and produces a summary report. Keep it simple — no frameworks, just clean Python modules. Suggest a folder structure and list each file with a one-line description of its purpose. Don't write any code yet.

The key moves: describe the project in plain English, set a constraint ("no frameworks"), and say "don't write any code yet" so you control the pacing. You can substitute any project idea you already have in mind.

2
Write
Generate code from a description
Ask AI to write a complete function from nothing but plain English

Open a new file called temperature.py. Open the chat panel and send the prompt below. Read what comes back before accepting — does it match what was asked? Is there anything you'd adjust?

Write a Python function called convert_temperatures that takes a list of temperatures in Celsius and returns a new list converted to Fahrenheit. Handle the case where the input list is empty. Add a short comment explaining the formula. No external libraries.

No starter code, no file context — just a description. Specific constraints ("no external libraries," "handle empty input") produce tighter, more predictable results than open-ended asks. Vague prompts get vague code.

3
Write
Finish code from a starter
Write the intention, let AI handle the implementation

Create a file called summarize.py and paste the code below exactly as written. Place your cursor at the end of the last comment line and press Enter. Wait — ghost text should appear suggesting the function body. Read it before pressing Tab to accept. If nothing appears after a few seconds, use the fallback prompt in the chat panel instead.

import pandas as pd def summarize_returns(filepath: str, ticker: str) -> dict: # Load a CSV with columns: date, ticker, close_price. # Filter rows for the given ticker. # Compute mean close price, max close, and min close. # Return results as a dictionary with keys: mean, max, min.

If Tab completion doesn't trigger, send this in the chat panel instead:

Complete the function body for summarize_returns based on the comments inside it. Use only pandas. Return the result as a plain dictionary.

The comment is doing the specification work. The more precise the comment, the more accurate the completion. Write the intention first — let AI fill in the implementation.

4
Understand
Explain a block of code
AI explaining code is often more useful than AI writing it

Create a file called pipeline.py and paste the function below into it. This function works, but it has some inefficiencies and a subtle bug hiding inside it. Select the entire function, open the chat panel, and send the prompt. Exercises 5 through 8 all use this same file.

import pandas as pd import time def process_sales(df: pd.DataFrame, year: int) -> pd.DataFrame: results = [] for i in range(len(df)): row = df.iloc[i] if row["year"] == year: margin = (row["revenue"] - row["cost"]) / row["revenue"] results.append({ "product": row["product"], "revenue": row["revenue"], "margin": margin }) summary = pd.DataFrame(results) summary = summary.sort_values("margin") total = 0 for i in range(len(summary)): total = total + summary.iloc[i]["revenue"] summary["revenue_share"] = summary["revenue"] / total return summary

Explain this function to me. Walk through what it does step by step, note any Python or pandas concepts worth understanding, and flag anything that looks inefficient or risky.

This works on any code you didn't write — library internals, inherited projects, examples found online. Asking AI to explain before asking it to change is a habit worth building.

5
Debug
Spot a bug
A second pair of eyes that has read every Python bug report ever written

The process_sales function from Exercise 4 has a subtle bug. Select the entire function in pipeline.py, open the chat panel, and send the prompt. See if the AI finds it before you do.

Review this function for correctness. Is there any input that would cause it to produce a wrong result or raise an error? Be specific about where the problem is and why it occurs.

The bug: when row["revenue"] is zero, the margin calculation — (row["revenue"] - row["cost"]) / row["revenue"] — divides by zero, producing inf or raising a ZeroDivisionError depending on the data type. It's the kind of mistake that only surfaces on real data, easy to overlook in a code review.

AI reviewing code is different from AI writing code. You're using it as a quality check — not to build something new, but to stress-test something that already exists. It catches this kind of structural mistake reliably.

6
Refactor
Improve performance and measure it
Give AI slow code, get fast code back — then prove it with a timer

The process_sales function loops over a DataFrame row by row — the slowest way to process data in pandas. Run the data generator below first to create a test file, then ask AI to rewrite the function. Finally, run the timing block to print both versions' runtimes to your terminal side by side.

Step 1 — Create the data. Paste this into a new file and run it once to generate sales_data.csv:

import pandas as pd import numpy as np df = pd.DataFrame({ "product": np.random.choice(["Widget", "Gadget", "Doohickey"], 50000), "year": np.random.choice([2022, 2023, 2024], 50000), "revenue": np.random.uniform(100, 10000, 50000).round(2), "cost": np.random.uniform(50, 8000, 50000).round(2), }) df.to_csv("sales_data.csv", index=False) print("sales_data.csv created — 50,000 rows")

Step 2 — Ask AI to refactor. Select the original process_sales function in pipeline.py and send this prompt:

Refactor this function to use vectorized pandas operations instead of iterating row by row. Fix the division-by-zero bug in the margin calculation while you're at it. Name the new function process_sales_fast. Keep the output identical — same columns, same structure.

Step 3 — Measure the difference. Add this timing block to the bottom of pipeline.py and run the file:

if __name__ == "__main__": df = pd.read_csv("sales_data.csv") start = time.perf_counter() process_sales(df.copy(), 2023) t_original = time.perf_counter() - start start = time.perf_counter() process_sales_fast(df.copy(), 2023) t_fast = time.perf_counter() - start print(f"Original: {t_original:.4f}s") print(f"Refactored: {t_fast:.4f}s") print(f"Speedup: {t_original / t_fast:.1f}x faster")

Expect a 10–50x speedup on 50,000 rows. The difference is not subtle. Vectorized pandas operations work on entire columns at once rather than one row at a time — AI knows every idiom for this and can apply them to any function immediately.

7
Annotate
Add type hints
Make your function's contract explicit in seconds

Select the process_sales_fast function from Exercise 6. Trigger inline edit and send the prompt below. A side-by-side diff will appear showing exactly what it proposes to change — review it before accepting.

Inline edit shortcut: Cmd+I on Mac  /  Ctrl+I on Windows

Inline edit shortcut: Ctrl+\  —  or right-click the selection and choose AI Actions

Add complete Python type hints to all parameters and the return value. Use specific types — not just Any or object.

Type hints don't change how the code runs — they make it readable and catchable. Once hints are in place, most editors will highlight type mismatches before you run anything.

8
Document
Generate a docstring
Professional Python code documents itself — AI writes the documentation

A docstring is a description block that lives inside a function, right below the def line, wrapped in triple quotes. Python uses it as built-in documentation — call help(process_sales_fast) in your terminal and it prints whatever the docstring says. Most professional Python code has them; most developers find writing them tedious. AI does not.

Select the refactored, type-hinted function from Exercise 7. Open the chat panel and send the prompt below.

Write a Google-style docstring for this function. Include a one-line summary, a description of each parameter with its type, and the return value. Keep it concise.

You can ask for different styles — NumPy and Sphinx are common alternatives. If your team has a documentation standard, specify it in the prompt and AI will follow it.

9
Test
Generate pytest cases
Ask AI to write tests for code it has already read

Make sure pipeline.py is open. Select the process_sales_fast function — with its type hints and docstring from the previous exercises — and open the chat panel. Send the prompt below.

Write three pytest functions for the process_sales_fast function I've selected — one normal case with typical data, one where no rows match the given year, and one where revenue is zero for some rows. Use only pytest and pandas. No fixtures.

Save the output to a new file called test_pipeline.py. Read each test before running it — make sure it's testing what it claims to test. Then run pytest test_pipeline.py -v in your terminal.

AI-generated tests are a starting point. They're useful for edge cases you might not have thought to write yourself — but they occasionally test the wrong thing or make incorrect assumptions about expected output. Read them before trusting them.

10
Share
Write a commit message from a diff
AI as a writing aid for the collaboration layer, not just the coding layer

No file needs to be open. Paste the diff below directly into the chat panel, then send the prompt. This simulates the moment just before committing — you have changes ready and want a clear, professional message that accurately describes what changed and why.

- def process_sales(df, year): + def process_sales_fast(df: pd.DataFrame, year: int) -> pd.DataFrame: + """Process sales data and compute margin and revenue share.""" # removed row-by-row iteration + filtered = df[df["year"] == year].copy() + filtered["margin"] = (filtered["revenue"] - filtered["cost"]) / \ + filtered["revenue"].replace(0, float("nan")) + filtered["revenue_share"] = filtered["revenue"] / filtered["revenue"].sum() + return filtered.sort_values("margin")

Write a concise git commit message for this diff. Use the conventional commits format: a type prefix, a short summary line, and one optional sentence of body if needed.

Conventional commits format: refactor: replace iterrows with vectorized operations. One line is usually enough. AI is good at distilling diffs into plain-English summaries — and consistent commit messages make project history readable by anyone who joins later.

Working well with AI

Learning to Prompt Well

AI will match whatever scope you give it. Ask for something small and focused, you get something small and focused. Ask something vague or enormous, and you'll get something vague or enormous — and probably lose the thread halfway through. The exercises above are deliberately contained for exactly this reason. The principles below are what make the difference between a productive session and an overwhelming one.

1

Give context before the ask

Tell AI what the project does, what role this code plays in it, and what you're trying to accomplish — before stating the specific request. Without context, AI fills in the gaps with assumptions, and those assumptions are often wrong in ways that are hard to spot. A sentence or two of setup changes the quality of the response significantly.

✕ "Write a function that cleans data."
✓ "I'm building a sales reporting script. I have a DataFrame with revenue and cost columns that sometimes contain nulls or negative values. Write a function that drops rows where either column is null or negative."
2

Constrain the scope explicitly

AI will produce as much as it thinks you want. Without limits, it defaults to comprehensive — which often means more than you need, more than you can review, and more than you can debug when something goes wrong. Explicit constraints keep the output reviewable. "Under 20 lines," "no external libraries," "just the function, no explanation," and "Python 3.11 only" are all useful constraints. Use them.

✕ "Add error handling to my project."
✓ "Add a try/except block to the load_csv function only. Catch FileNotFoundError and print a clear message. Don't touch anything else."
3

Ask for a plan before you ask for code

For anything larger than a single function, ask AI to describe its approach before writing anything. This is what "don't write any code yet" does in Exercise 1 — it forces the AI to think out loud and gives you a chance to redirect before you're buried in output you don't want. Once you've agreed on the approach, asking for the code produces something much closer to what you actually had in mind.

✓ "Before writing any code, describe how you'd structure a function that does X. What edge cases would you handle?"
4

Tell it your level and what you want to learn

AI adapts its output significantly based on who it thinks is reading. Saying "I'm new to pandas" produces simpler methods and more explanation. Saying "explain your reasoning alongside the code" gets you something you can learn from rather than just copy. If you want to understand what it built — ask. That's a separate prompt, and a valuable one.

✓ "I'm learning pandas. Write this using basic operations and add a comment on each line explaining what it does."
Example prompt
// context — what the project is and what this code needs to do I'm building a Python script that reads monthly sales CSVs and produces a summary report. Each CSV has columns: product, region, revenue, and cost. I need a function that loads one CSV, drops rows where revenue or cost is missing, and returns a cleaned DataFrame. // constraints — scope, libraries, style Keep it under 20 lines. Use only pandas and the standard library. No classes — just a single function. // plan before code Before writing the function, describe how you'd handle the case where the file doesn't exist or is empty. Then write the code. // level and learning intent I'm comfortable with Python basics but newer to pandas. Add a short comment on any pandas method that isn't obvious.

When you feel lost or overwhelmed

Getting a response that's too long, too complex, or just not what you wanted is feedback about the prompt — not about your ability to code. These three moments come up often, and each one has a direct fix.

👉
The response is longer or more complex than you expected

Push back directly in the same chat: "That's more than I need — simplify it. I just want the core logic, no extra abstractions." AI will revise. You don't need to start over or feel like you asked the wrong question. Simplification is a completely normal follow-up request, and AI handles it well.

🔁
You've lost the thread mid-conversation

Long conversations accumulate context that can start working against you — AI tries to stay consistent with earlier messages, even when you've changed direction. If a conversation feels tangled, start a fresh chat and re-state what you actually need now. "Ignore the previous approach. Let's start over — I need something simpler." A clean slate is faster than untangling.

⚠️
AI produced something that looks right but might not be

Confidence is not correctness. AI can write fluent, well-structured code that has a subtle logic error, makes a wrong assumption about your data, or uses an API slightly incorrectly. Running the code is always the check — not reading it and assuming it's fine. When something fails, paste the exact error back into the chat rather than trying to interpret it yourself first.

🌍
Your editor, your choice

VS Code with GitHub Copilot and PyCharm with JetBrains AI Assistant are the most common starting points and what this page references. But they are not the only options. Cursor is a fork of VS Code rebuilt with in-context, inline AI editing as its core purpose rather than a plugin layer — popular and worth a look once you're comfortable with the fundamentals. Zed, Windsurf, and others follow the same idea.

📄
How file context works

When you open a file and open the chat panel, the AI has already read your open file before you type anything. You can also reference files explicitly — in VS Code Copilot Chat, type #filename.py in your message. In PyCharm AI Assistant, the open file is included automatically and you can drag additional files into the chat. This is what makes in-editor AI different from browser chat: it is not guessing about your code. It has read it.

What's next

Level 3 — AI in your terminal

Level 2 keeps AI inside the editor. It reads your files and proposes changes, but it cannot run anything — you still execute code in the terminal, read the output, and carry results back into the chat manually.

Level 3 removes that constraint. A terminal agent like Claude Code lives in your command line and has access to your full computing environment: it can execute scripts, read the output, install packages, write and modify files anywhere, and iterate on its own until a task is complete. You describe an outcome; the agent handles the execution loop. The IDE is still useful for reviewing what it produced, but it is no longer required to drive the task.
